AVERA is a local engineering tool that takes your existing verification artifacts, compares baseline versus current results, maps impact to requirements, and produces a structured release decision — backed by explicit evidence, not interpretation.

1 · What it is

The problem it solves

Every automotive engineering team already has requirements, tests, verification runs, logs, and change history. What is often missing is a reliable way to connect those artifacts into one proof chain before release.

When a change is made, the team needs to answer four questions quickly:

  • Is this failure new or was it already there before the change?
  • Which requirements does the failure actually touch?
  • Is the evidence strong enough to justify a release block?
  • Who needs to act, and what exactly should they verify next?

AVERA answers those four questions from your existing artifacts — without guessing, without summarizing vaguely, and without losing the evidence chain when the result is handed to another reviewer.

2 · How it works
Step 1
You provide your artifacts
Change description, baseline verification results, current verification results, requirements export, and component map. All local files — no upload, no account.
Step 2
AVERA runs the analysis
One command. AVERA compares baseline vs current, maps failures to requirements, classifies risk conservatively, and evaluates whether the evidence justifies a release block.
Step 3
You get a structured result
A verdict, confidence score, affected requirements list, threshold evidence, decision record, and a portable workspace pack for handoff — all generated locally.
What you put in
  • Change description .txt
  • Baseline results baseline_results.json
  • Current results current_results.json
  • Requirements requirements.csv
  • Component map component_map.json
  • Signal trace (optional) signal_trace.csv
What comes out
  • Report avera-report.json / .md
  • Evidence graph avera-evidence-graph.json
  • Gate result — pass / review / block
  • Traceability index
  • Decision record with owner routing
  • Workspace pack for handoff

5 · What it gives you

Without AVERA

  • You see that a test failed — but not whether it was already failing before the change
  • You know something is wrong — but not which requirements are actually affected
  • You have a feeling it should be blocked — but not a structured decision with evidence attached
  • You explain the issue to a reviewer — and they have to reconstruct the context from scratch

With AVERA

  • New failure vs preexisting failure — distinguished automatically from baseline comparison
  • Requirement impact — mapped from changed file through component to affected requirements
  • Release decision — structured as block / review / pass with confidence score and rationale
  • Handoff artifact — workspace pack contains the full evidence chain, ready for any reviewer
